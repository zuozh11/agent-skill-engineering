# Prototype

原型是**回答一个问题的一次性代码**。问题决定形态。

## 选择分支

识别正在回答哪个问题——从用户的提示、周围的代码，或在用户在线时直接询问：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个小型交互式终端应用，将状态机推过那些在纸上难以推理的场景。
- **"这个应该长什么样？"** → [UI.md](UI.md)。在单个路由上生成几个根本不同的 UI 变体，通过 URL 搜索参数和浮动底栏切换。

两个分支产出非常不同的制品——选错了整个原型就白费了。如果问题确实模糊且用户不可达，默认选择与周围代码更匹配的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部说明假设。

## 两个分支通用的规则

1. **从第一天起就是一次性的，并且明确标记。** 将原型代码放在它实际使用位置的附近（它所原型验证的模块或页面旁边），这样上下文显而易见——但命名要让随意的读者能看出这是原型，不是生产代码。对于一次性 UI 路由，遵循项目已有的路由约定；不要发明新的顶层结构。
2. **一条命令运行。** 项目现有任务运行器支持的任何方式——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须不用思考就能启动它。
3. **默认不持久化。** 状态存在内存中。持久化是原型正在*检验*的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，使用临时数据库或带有明确"PROTOTYPE——可清除"名称的本地文件。
4. **跳过打磨。** 不写测试，不做超出让原型*可运行*所需的错误处理，不做抽象。重点是快速学到东西然后删除它。
5. **展示状态。** 每次操作（逻辑）或每次变体切换（UI）后，打印或渲染完整的相关状态，让用户看到什么变了。
6. **完成后删除或吸收。** 当原型回答了它的问题后，要么删除它，要么将验证过的决策融入真实代码——不要让它在仓库里腐烂。

## 完成时

原型中唯一值得保留的是*答案*。将它记录在持久的地方（commit 消息、ADR、issue 或原型旁边的 `NOTES.md`），连同它回答的问题。如果用户在线，记录就是一次简短对话；如果不在，留下占位符以便他们（或你，在下一轮）在删除原型前填入结论。
