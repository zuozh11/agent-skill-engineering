# Impl

根据任务卡或当前对话实现代码变更，两种输入方式地位相同。

**原子提交是本 Skill 最核心的原则：一个独立实现单元 = 一个 commit。**

## 流程

### 1. 确定实现单元

从用户提供的输入中确定本次要完成的独立实现单元：

- **任务卡输入**：读取指定的 task 文件，理解目标、现状、方案和验收条件。
- **对话输入**：从当前对话提取目标、范围、已确定的方案和验收方式；上下文足够时直接实现，不要求先生成任务卡。

如果一次包含多个可独立完成的改动，先划分实现单元，后续分别验证和提交。只有缺少的信息会真正阻塞实现时，才调用 `grill-with-docs` 与用户对齐。

### 2. 加载领域知识与编码约束

按需查阅以下资源：

- **领域文档**：按 `docs/agents/domain.md` 定位并读取相关 `CONTEXT.md` 与 RULES。只要某条 RULES 有哪怕约 1% 的可能影响当前任务，就必须读取；只有明确完全无关时才可跳过。相关文档不存在则静默继续。

### 3. 安排执行方式

根据实现单元数量、依赖关系、写集冲突和上下文规模，自行决定直接实现还是编排子 Agent，不需要用户确认。

- 单个实现单元、强依赖或预计修改同一批文件时，直接在主会话按顺序实现。
- 多个相互独立、适合隔离上下文的实现单元，交给子 Agent；无依赖、无写集冲突时并行执行，其余按顺序执行。
- 并行子 Agent 必须避免修改互相污染；无法保证时改为串行。
- 子 Agent 必须收到完整的实现目标；存在 task 文件时同时提供文件路径和内容，并附上 `CONTEXT.md` 指针、适用 RULES 摘要，以及执行步骤 4-6 并原子提交的指令。

---

### 4. 实现

- 严格遵守步骤 2 识别的相关 RULES；存在 task 文件时，同时读取其中列出的适用 RULES
- 根据任务卡或对话确定的目标查看现有代码，将已确定的方案映射到实际代码结构，并识别必要的修改点
- 不重新选择任务卡或对话中已经收口的主要方案；若代码事实证明方案不可行或互相矛盾，停止实现并明确报告，而不是自行改换方案
- 遵守 `CLAUDE.md` / `AGENTS.md` 中的编码指南和规则
- 匹配实现范围内相关的 Skill
- 不添加实现范围之外的功能、重构或抽象

### 5. 验证

- 对照任务卡中的验收条件或对话中确定的目标逐条自检
- 对照步骤 2 已读取的 RULES 逐条核对实现无违反；如违反，按"默认遵守"修复后重新验证，确需偏离则显式标出交用户裁决
- 如果验证失败，修复后重新验证
- 存在 task 文件时，验证通过后把 `Status:` 从 `ready-for-agent` 改为 `completed`，并勾选已通过的验收项

### 6. 原子提交（核心）

每个实现单元验证完成后，立即将对应的代码、测试和配置改动提交为一个 commit。

一次包含多个实现单元时，必须分别形成独立 commit，不合并提交，不攒到最后。

提交内容包含：
- 当前实现单元对应的代码、测试和配置改动
- 存在 task 文件时，同时包含该文件的状态更新

不要把其它实现单元的改动或状态更新带进同一个 commit。

## 硬性规则

- **不扩展 scope**：只做任务卡或对话确定的改动，发现相邻问题只提醒不修
- **不过度抽象**：匹配现有代码风格，不引入任务未要求的设计模式
- **积极复用**：优先复用现有能力、组件和同类实现，不重复实现已有逻辑，也不为复用扩大实现范围
- **不猜 API**：不确定的框架 API 先 grep 现有代码中的用法，再查 references
- **不跳过状态更新**：存在 task 文件时，验证通过后先标为 `completed`，再把实现和任务状态一起提交
