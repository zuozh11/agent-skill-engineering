# Impl

按任务卡实现代码变更的通用 skill。一个 task = 一次实现 = 一个原子提交。

## 流程

### 1. 读取任务

读取用户指定的 task 文件（`docs/scratch/<feature-slug>/tasks/NN-xxx.md`）。理解：
- What to build（端到端行为）
- 改动文件（路径 + 现状）
- 改动内容（字段清单 / 伪码片段 / 代码模板）
- 验收条件

如果用户没有指定文件而是口头描述任务，按描述执行，但仍遵循下面的约束加载流程。如果口头描述的上下文不足以直接执行，调用 `grill-with-docs` 与用户对齐。

### 2. 加载领域知识与编码约束

按需查阅以下资源：

- **领域文档**（`CONTEXT.md` + `docs/rules/`）：默认读 `docs/CONTEXT.md` + `docs/rules/`（按文件名挑相关的读，拿不准就读；不存在则静默继续）。若存在 `docs/CONTEXT-MAP.md`（多 Context monorepo），规则分系统级 + Context 级**两层**、别只读根目录，定位相关 Context 与落点见 `docs/agents/domain.md`。

### 3. 选择执行方式（≥2 个 task 时触发）

单任务直接跳到步骤 4。多任务时由本 skill **自主决定**执行方式，**不询问用户**：

- **内联执行**：仅当任务极少（2~3 个）、单个简单、且构成强依赖链或预计会改动同一批文件时采用。在当前上下文逐个执行步骤 4-7，逐个原子提交。
- **编排执行**（多任务默认）：其余情况一律走编排，把每个 task 交给独立子 Agent——隔离上下文、结构化追踪、可恢复，并在没有预计冲突时并行推进。

编排按工具可用性二选一，判定要稳健：**检查当前环境是否提供 Workflow 编排工具**，提供则用 Workflow，否则回退到 fork 子 Agent。两条路径都先根据任务卡中的 `Blocked by`、改动文件清单和验收范围建立依赖 / 冲突图：

- `Blocked by` 未完成的 task 不启动。
- 预计会写同一文件、同一配置、同一生成物、同一 task 状态文件，或需要共享迁移 / 先后顺序的 task 视为冲突，必须排序执行。
- 没有依赖关系、没有预计写冲突、验收可独立完成的 task **应该并行执行**。

> **默认并行（关键约束）**：编排执行不是串行 `for` 循环。只要任务之间没有预计冲突，就同时 fork 多个子 Agent 并发执行；只有存在明确依赖或预计冲突时才降级为串行。

> **原子提交保护**：并行子 Agent 不得共享同一个可写工作树直接提交。并行实现时优先为每个 task 使用独立 worktree / 隔离环境；如果当前工具无法隔离可写工作树，只有在确认不会争抢 git index、不会污染彼此提交时才并行，否则把冲突组串行化。每个 task 仍必须独立完成步骤 4-7 并生成一个原子提交。

每起一个子 Agent，prompt 包含：

- task 文件完整内容
- 领域知识指针（`CONTEXT.md`、相关 `RULES` 的路径；具体读取逻辑见步骤 2）
- 明确指令：执行步骤 4-7，完成后原子提交

按依赖 / 冲突图推进：同一批可并行的 task 同时启动；任一 task 完成并提交后，重新检查被它解除依赖的 task，继续启动下一批无冲突任务。

---

### 4. 实现

- 严格遵守步骤 2 识别的相关 RULES——这些是项目硬约束（命名/单位/错误处理/日志等），与现有代码或习惯冲突时以 RULES 为准
- 先查看任务涉及文件的现有代码（grep 同类实现、读取目标文件）
- 使用 CONTEXT.md 中的术语命名变量和方法
- 遵守 `CLAUDE.md` / `AGENTS.md` 中的编码指南和规则
- 匹配 任务范围内相关的 skill
- 不添加任务范围之外的功能、重构或抽象

### 5. 验证

- 对照任务验收条件逐条自检
- 对照本任务涉及的 RULES 逐条核对实现无违反；如违反，按"默认遵守"修复后重新验证，确需偏离则显式标出交用户裁决
- 如果验证失败，修复后重新验证
- 验证通过后，把当前 task 的 `Status:` 从 `ready-for-agent` 改为 `completed`
- 已验证通过的验收项同步勾选，保持任务卡和实现状态一致

### 6. 原子提交

一个 task = 一个 commit。

如果一次接到多个 task，按依赖顺序逐个执行；每完成一个 task 都必须立即提交，不攒到最后。

提交内容包含：
- 当前 task 文件的状态更新
- 当前 task 对应代码 / 测试 / 配置改动

不要把其它未完成 task 的状态更新带进同一个 commit。

## 硬性规则

- **不扩展 scope**：只做任务卡描述的改动，发现相邻问题只提醒不修
- **不过度抽象**：匹配现有代码风格，不引入任务未要求的设计模式
- **不猜 API**：不确定的框架 API 先 grep 现有代码中的用法，再查 references
- **不跳过状态更新**：实现验证通过后，先把当前 task 标为 `completed`，再把实现和任务状态一起提交
