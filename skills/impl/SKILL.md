# Impl

按任务卡实现代码变更的通用 skill。一个 task = 一次实现 = 一个原子提交。

## 流程

### 1. 读取任务

读取用户指定的 task 文件（`docs/scratch/<feature>/tasks/NN-xxx.md`）。理解：
- What to build（端到端行为）
- 改动文件（路径 + 现状）
- 改动内容（字段清单 / 伪码片段 / 代码模板）
- 验收条件

如果用户没有指定文件而是口头描述任务，按描述执行，但仍遵循下面的约束加载流程。

### 2. 加载领域知识与编码约束

按需加载，不要一次性全读：

| 需要了解           | 去哪里看                                  |
|----------------|---------------------------------------|
| 业务术语、实体关系      | `CONTEXT.md`                          |
| 架构硬约束          | `docs/adr/`（只读与当前改动相关的）                |

### 3. 实现

- 先查看任务涉及文件的现有代码（grep 同类实现、读取目标文件）
- 使用 CONTEXT.md 中的术语命名变量和方法
- 遵守 ADR 中的约束
- 遵守 `CLAUDE.md` / `AGENTS.md` 中的编码指南和规则
- 匹配 任务范围内相关的 skill
- 不添加任务范围之外的功能、重构或抽象

### 4. 验证

- 对照任务验收条件逐条自检
- 如果验证失败，修复后重新验证
- 验证通过后，把当前 task 的 `Status:` 从 `ready-for-agent` 改为 `completed`
- 已验证通过的验收项同步勾选，保持任务卡和实现状态一致

### 5. 原子提交

一个 task = 一个 commit。

如果一次接到多个 task，按依赖顺序逐个执行；每完成一个 task 都必须立即提交，不攒到最后。

提交内容包含：
- 当前 task 文件的状态更新
- 当前 task 对应代码 / 测试 / 配置改动

不要把其它未完成 task 的状态更新带进同一个 commit。

## 硬性规则

- **不扩展 scope**：只做任务卡描述的改动，发现相邻问题只提醒不修
- **不过度抽象**：匹配现有代码风格，不引入任务未要求的设计模式
- **不猜 API**：不确定的框架 API 先 grep 现有代码中的用法，再查 references
- **不跳过状态更新**：实现验证通过后，先把当前 task 标为 `completed`，再把实现和任务状态一起提交
