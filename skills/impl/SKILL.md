# Impl

按任务卡实现代码变更的通用 skill。一个 task = 一次实现 = 一个原子提交。

## 流程

### 1. 读取任务

读取用户指定的 task 文件（`docs/scratch/<feature>/tasks/NN-xxx.md`）。理解：
- What to build（端到端行为）
- 改动文件（路径 + 现状）
- 改动内容（字段清单 / 伪码片段 / 代码模板）
- 验收条件

如果用户没有指定文件而是口头描述任务，按描述执行，但仍遵循下面的约束加载流程。如果口头描述的上下文不足以直接执行，调用 `grill-with-docs` 与用户对齐。

### 2. 加载领域知识与编码约束

按需查阅以下资源：

- 仓库根目录的 **`CONTEXT.md`**
- **`docs/adr/`** — 强制读取全部架构决策记录（ADR），无需判断相关性，每次任务开始前直接全部读取

### 3. 评估开发模式（≥2 个 task 时触发）

单任务直接跳到步骤 4。多任务时评估以下维度，选择开发模式：

**评估维度**：
- 任务数量和复杂度
- 依赖关系（`Blocked by` 字段）
- 写集冲突风险（共享文件的重叠程度）

**三种模式**：

| 模式 | 适用场景 | 执行方式 |
|------|----------|----------|
| A. 顺序直接开发 | 2-3 个简单任务，强依赖链 | 当前上下文逐个执行步骤 4-7 |
| B. 子 Agent 顺序开发 | 任务较复杂或数量多，有依赖链 | fork agent 逐个执行，隔离上下文 |
| C. 子 Agent 并行开发 | 任务间无依赖或可分组并行 | fork agent + worktree 并行执行 |

用`提问工具`展示推荐模式及理由，让用户最终决定。

**推荐优先级**：当 fork 子代理可用时，模式 B 的推荐程度始终高于模式 A（上下文隔离收益大于切换开销）。仅当任务极简且总数 ≤2 时才考虑模式 A。

---

选择模式 A 后按步骤 4-7 顺序执行。选择模式 B/C 则进入下方编排流程，步骤 4-7 由 fork agent 内部执行。

#### 模式 B — 子 Agent 顺序开发

按依赖顺序，每个 task fork 一个 agent（1 task = 1 agent，一对一）：

1. 按 `Blocked by` 拓扑排序，确定执行顺序
2. 对当前 task fork agent，prompt 包含：
   - task 文件完整内容
   - 领域知识路径（CONTEXT.md、相关 ADR）
   - 明确指令：执行步骤 4-7，完成后原子提交
3. 等待 fork agent 完成并确认提交成功
4. 进入下一个 task，重复步骤 2-3

#### 模式 C — 子 Agent 并行开发

1. **构建依赖图**：解析所有 task 的 `Blocked by` 字段
2. **分层调度**：拓扑排序后分为可并行的"层"（同层内任务互不依赖、无写集冲突）
3. **逐层执行**：
   - 为当前层的每个 task 创建独立 worktree（基于当前分支）
   - 同层所有 task 同时 fork agent（1 task = 1 agent），每个 agent 在自己的 worktree 中执行步骤 4-7
   - 等待同层全部 agent 完成
4. **合并**：按顺序将各 worktree 分支合并回当前分支
   - 如出现冲突，由编排者（当前上下文）手动解决
5. **推进下一层**：基于合并后的分支创建新 worktree，重复步骤 3-4
6. **清理**：所有层完成后删除所有 worktree

---

### 4. 实现

- 先查看任务涉及文件的现有代码（grep 同类实现、读取目标文件）
- 使用 CONTEXT.md 中的术语命名变量和方法
- 遵守 ADR 中的约束
- 遵守 `CLAUDE.md` / `AGENTS.md` 中的编码指南和规则
- 匹配 任务范围内相关的 skill
- 不添加任务范围之外的功能、重构或抽象

### 5. 实现后复审

代码改完、未提交前调用 `advisor` 或 `review agent` 对实现做一次复审：方向是否符合任务卡、是否引入任务范围外的改动、是否触碰任务卡中"明确不做"的边界。按反馈修订实现。

### 6. 验证

- 对照任务验收条件逐条自检
- 如果验证失败，修复后重新验证
- 验证通过后，把当前 task 的 `Status:` 从 `ready-for-agent` 改为 `completed`
- 已验证通过的验收项同步勾选，保持任务卡和实现状态一致

### 7. 原子提交

一个 task = 一个 commit。

如果一次接到多个 task，按依赖顺序逐个执行；每完成一个 task 都必须立即提交，不攒到最后。

提交内容包含：
- 当前 task 文件的状态更新
- 当前 task 对应代码 / 测试 / 配置改动

不要把其它未完成 task 的状态更新带进同一个 commit。

## 硬性规则

- **不扩展 scope**：只做任务卡描述的改动，发现相邻问题只提醒不修
- **不过度抽象**：匹配现有代码风格，不引入任务未要求的设计模式
- **不猜 API**：不确定的框架 API 先 grep 现有代码中的用法，再查 references
- **不跳过状态更新**：实现验证通过后，先把当前 task 标为 `completed`，再把实现和任务状态一起提交
