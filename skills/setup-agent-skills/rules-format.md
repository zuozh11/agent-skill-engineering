# RULES Format

RULES 是项目长期有效的规则、约定与约束——agent 在相关任务中必须查阅并遵守，否则实现会跑偏。它既包含难逆转的关键架构决策，也包含 agent 实现时必须对齐的项目约定（命名、单位、错误处理、日志规范等）。它和 `CONTEXT.md` 不同：CONTEXT.md 记录业务术语，RULES 记录项目规则，两者基本不重叠。

RULES 存放在 `docs/rules/` 目录下，每条规则一个文件。`docs/rules/` 目录按需创建——只在需要第一条规则时才建。

## 命名

文件名 = 两位顺序编号 + 规则主题的中文短语：`NN-<规则主题>.md`，自描述，一眼知道这条规则管什么。

- 编号给一个稳定的短引用，主题让人一眼看懂规则管什么。
- 主题直接描述规则管的事，不必套用 `CONTEXT.md` 的业务术语——RULES 是项目通用规则，与业务术语表基本不重叠。
- 示例：
  - `01-数据权限-按部门隔离.md`
  - `02-用户信息-从登录态获取.md`
  - `03-接口错误码-统一包装.md`
  - `04-模块通信-领域事件优先.md`
- 单 Context 与多 Context 使用同一命名方式。多 Context 仓库的规则也只存放在根 `docs/rules/`，不添加 Context 前缀；如需限定适用范围，在规则正文中说明。
- 相互引用时用短号 `RULE-NN`（如 `RULE-03`）或《主题》；commit / PR 里推荐用短号。

## 编号

在根 `docs/rules/` 目录内扫描最大的现有编号，加一。两位零填充（`01`、`02`……）。无论单 Context 还是多 Context，整个仓库共用这一条编号序列。

## 模板

```md
# {规则主题的简短标题}

{1-3 句：定了什么规则，以及（如有）为什么。}
```

就这些。一条规则可以只是一段话。价值在于把规则明确写下来供 agent 遵守——决策类规则要讲清"为什么这么定"，约定类规则的"为什么"可以很简短甚至省略，关键是把约定固化下来。

## 可选章节

只在确实有价值时才包含这些。大多数规则不需要它们。

- **Status** 前置元数据（`proposed | accepted | deprecated | superseded by 《其他规则》`）——在规则被重新审视时有用
- **Considered Options** ——只在被否决的替代方案值得记住时才写
- **Consequences** ——只在非显而易见的下游影响需要指出时才写

## 何时提议一条 RULES

RULES 收录两类规则，满足其一即应记录：

**A. 决策类**（架构形态、技术选型、模块边界等"为什么这么定"的关键决策）——三个条件同时满足：

1. **难以逆转** ——事后改变主意的代价很大
2. **缺少上下文会让人困惑** ——未来的读者看到代码会想"他们为什么要这么做？"
3. **真正的权衡结果** ——确实存在多个替代方案，你基于具体原因选择了其中一个

**B. 约定/约束类**（错误码统一包装、金额以分为单位、日志规范、命名约定等 agent 必须对齐的项目惯例）——只需满足：

- **项目长期有效** ——是跨任务复用的约定，不是一次性或局部实现细节
- **agent 必须遵守，违反就会跑偏** ——不写下来，不同实现就会各写各的、产生不一致

约定类**不要求**难逆转或多方案权衡——它的价值不在"解释取舍"，而在"提供 agent 必须遵守的统一约束"。

反之，符合以下任一就跳过：一次性的、局部实现细节、能从代码一眼看出的、显而易见无异议的。

### 什么情况适合记录

- **架构形态。** "我们使用 monorepo。""写模型是事件溯源的，读模型投影到 Postgres。"
- **上下文之间的集成模式。** "Ordering 和 Billing 通过领域事件通信，而非同步 HTTP。"
- **带有锁定效应的技术选型。** 数据库、消息总线、认证提供商、部署目标。不是每个库——只是那些需要一个季度才能替换的。
- **边界和范围决策。** "客户数据归 Customer 上下文所有；其他上下文仅通过 ID 引用。"明确的"不做"和"做"一样有价值。
- **刻意偏离显而易见路径的决策。** "我们用手写 SQL 而不是 ORM，因为 X。"任何合理读者会假设相反做法的地方。这能阻止下一个工程师"修复"一个本来就是刻意为之的东西。
- **代码中不可见的约束。** "由于合规要求我们不能使用 AWS。""响应时间必须低于 200ms，因为合作伙伴 API 合约的要求。"
- **被否决的替代方案（当否决原因不明显时）。** 如果你考虑过 GraphQL 但因为微妙的原因选择了 REST，记录下来——否则六个月后有人会再次提议 GraphQL。
- **跨代码强制的项目约定。** agent 实现时必须统一遵守、否则会不一致的规范："接口错误码统一包装成 {code, message, data}""金额一律以分为单位存储""日志必须带 traceId""数据库表名用复数 snake_case"。这类约定不一定难逆转、也无多方案权衡，但 agent 不知道就会写错。
