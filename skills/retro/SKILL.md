# Retro

复盘 review 修改，闭合工程管线的反馈环。

## 调用方式

```
/retro [range]
```

range 可选。如果未指定，默认使用**当前会话**中的变更作为分析范围。

## 流程

### 1. 确定分析范围

默认范围：当前会话中产生的变更（对话上下文 + 工作区 diff）。

如果用户指定了 commit range，则从该范围获取变更：

- `git diff <range>` — 已提交的变更
- `git diff` + `git diff --staged` — 未提交的改动（一并纳入分析）

### 2. 加载上下文

| 来源 | 用途 |
|------|------|
| 当前会话对话 | 提取 review 讨论中的认知盲区（B 层） |
| task 文件 | 对比预期 vs 实际（A 层） |
| PRD | 理解原始需求意图 |
| docs/rules/ | 检查是否有需要补充的项目规则（RULES）；按 docs/agents/domain.md 布局读取，多 Context 下含系统级与相关 Context 级两层，两层都要读、别只读根目录 |

如果 task 文件或 PRD 不存在（用户直接从口头描述开始 impl），跳过 A 层对比，只做 B 层分析。

### 3. A 层分析：对话中的纠偏

重点分析当前会话中用户指出的问题和纠正：

- 用户明确否定了哪些做法？原因是什么？
- 用户要求返工或修改的部分，暴露了哪些认知偏差？
- 对比 task 文件预期 vs 实际产出，偏差是 review 驱动的还是实现中跑偏的？

### 4. B 层分析：认知盲区

从当前会话的 review 对话中提取：

- reviewer 指出了哪些 grill/PRD 阶段没考虑到的问题？
- 哪些修改暴露了 RULES 中未覆盖的项目规则——可能是架构约束，也可能是项目约定（命名、错误处理、单位、日志等）？
- 哪些修改是 agent 反复犯的模式性错误？

### 5. 分类与标注

将每条发现分类为两种产物之一：

| 分类   | 触发条件                  | 产物              |
|------|-----------------------|-----------------|
| 需求补充 | PRD 中有遗漏或歧义           | PRD.md 更新       |
| 项目规则 | review 暴露了 agent 未对齐的项目规则——架构约束、集成模式，或纯约定/约束（错误码包装、单位、日志、命名等） | docs/rules/ 新增 RULES |

为每条发现标注置信度：
- **高**：明确的模式性问题，大概率会反复出现
- **中**：有一定通用性，但可能是特定场景
- **低**：可能是一次性问题，但值得记录

### 6. 批量审批

将所有发现以清单形式呈现给用户，每条包含：

- 分类标签（需求补充 / 项目规则（RULES））
- 置信度（高 / 中 / 低）
- 内容摘要
- 建议的 draft 内容

使用 `提问工具` 让用户批量审批，确认后写入对应文件。

### 7. 写入

根据用户审批结果执行写入：

**PRD 更新**：如果复盘发现原始需求描述有遗漏或歧义，更新对应的 PRD.md。

**项目规则**：写入 `docs/rules/` 下新的 RULES 文件（命名见 docs/agents/rules-format.md；多 Context 下落系统级还是相关 Context 级见 docs/agents/domain.md）。

## 硬性规则

- **不自动写入**：所有产物必须经过用户审批才能写入
- **不编造偏差**：如果 review 修改合理且无认知盲区，如实报告"无显著发现"
