# 术语

本 skill 所有建议使用的共享词汇。请精确使用这些术语——不要替换为"组件"、"服务"、"API"或"边界"。一致的术语就是全部要点。

## 术语

**模块（Module）**
任何具有接口和实现的东西。刻意与规模无关——同样适用于函数、类、包或跨层切片。
_避免使用_: unit, component, service。

**接口（Interface）**
调用者正确使用模块所必须知道的一切。包括类型签名，但也包括不变量、顺序约束、错误模式、所需配置和性能特征。
_避免使用_: API, signature（太窄——它们只指类型层面的表面）。

**实现（Implementation）**
模块内部的东西——它的代码体。与**适配器**不同：一个东西可以是小适配器大实现（Postgres 仓储），也可以是大适配器小实现（内存 fake）。当话题是接缝时用"适配器"；否则用"实现"。

**深度（Depth）**
接口处的杠杆——调用者（或测试）每学习一个单位的接口就能行使的行为量。当大量行为藏在小接口后面时，模块是**深的**。当接口几乎和实现一样复杂时，模块是**浅的**。

**接缝（Seam）** _（来自 Michael Feathers）_
一个你可以在不编辑该处代码的情况下改变行为的地方。模块接口所在的*位置*。选择接缝放在哪里本身就是一个设计决策，与接缝后面放什么是分开的。
_避免使用_: boundary（与 DDD 的限界上下文重载）。

**适配器（Adapter）**
在接缝处满足接口的具体事物。描述的是*角色*（它填充什么槽位），而不是实质（里面是什么）。

**杠杆（Leverage）**
调用者从深度中获得的东西。每学习一个单位的接口就获得更多能力。一个实现在 N 个调用点和 M 个测试中回报。

**局部性（Locality）**
维护者从深度中获得的东西。变更、bug、知识和验证集中在一个地方，而不是分散在调用者之间。修一次，处处修好。

## 原则

- **深度是接口的属性，不是实现的属性。** 一个深模块内部可以由小的、可 mock 的、可替换的部分组成——它们只是不属于接口的一部分。一个模块可以有**内部接缝**（实现私有的，被自己的测试使用）以及接口处的**外部接缝**。
- **删除测试。** 想象删除这个模块。如果复杂性消失了，这个模块没有隐藏任何东西（它只是一个透传）。如果复杂性重新出现在 N 个调用者中，这个模块在发挥作用。
- **接口就是测试面。** 调用者和测试跨越同一个接缝。如果你想测试接口*之后*的东西，模块的形状可能不对。
- **一个适配器意味着假设性接缝。两个适配器意味着真实接缝。** 除非确实有东西在接缝两侧变化，否则不要引入接缝。

## 关系

- 一个**模块**恰好有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是**模块**的属性，相对于其**接口**来衡量。
- **接缝**是**模块**的**接口**所在的位置。
- **适配器**位于**接缝**处并满足**接口**。
- **深度**为调用者产生**杠杆**，为维护者产生**局部性**。

## 被否决的框架

- **深度作为实现行数与接口行数的比率**（Ousterhout）：奖励填充实现。我们改用深度即杠杆。
- **"接口"作为 TypeScript `interface` 关键字或类的公共方法**：太窄——这里的接口包括调用者必须知道的每一个事实。
- **"边界（Boundary）"**：与 DDD 的限界上下文重载。说**接缝**或**接口**。
