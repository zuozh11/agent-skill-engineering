# 加深

如何在给定依赖关系的情况下安全地加深一组浅模块。假设使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块**、**接口**、**接缝**、**适配器**。

## 依赖分类

评估加深候选时，对其依赖进行分类。分类决定了加深后的模块如何跨接缝进行测试。

### 1. 进程内

纯计算，内存状态，无 I/O。始终可加深——合并模块并直接通过新接口测试。不需要适配器。

### 2. 本地可替代

具有本地测试替身的依赖（PGLite 替代 Postgres，内存文件系统）。如果替身存在则可加深。加深后的模块在测试套件中使用替身运行测试。接缝是内部的；模块的外部接口不暴露端口。

### 3. 远程但自有（端口与适配器）

跨网络边界的自有服务（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深模块拥有逻辑；传输层作为**适配器**注入。测试使用内存适配器。生产使用 HTTP/gRPC/队列适配器。

建议形式：*"在接缝处定义一个端口，为生产实现 HTTP 适配器，为测试实现内存适配器，这样逻辑就在一个深模块中，即使它跨网络部署。"*

选择传输/适配器与跨上下文集成方式前，查阅 docs/rules/ 中的相关规则（按 docs/agents/domain.md 布局定位，多 Context 含系统级与相关 Context 级两层）——集成方式、消息总线、外部依赖常被项目 RULES 约束，加深建议须落在这些约束内。

### 4. 真正的外部（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。加深后的模块将外部依赖作为注入的端口；测试提供 mock 适配器。

## 接缝纪律

- **一个适配器意味着假设性接缝。两个适配器意味着真实接缝。** 除非至少有两个适配器是合理的（通常是生产 + 测试），否则不要引入端口。单适配器接缝只是间接层。
- **内部接缝 vs 外部接缝。** 深模块可以有内部接缝（实现私有的，被自己的测试使用）以及接口处的外部接缝。不要仅仅因为测试使用了内部接缝就通过接口暴露它们。

## 测试策略：替换，不要叠加

- 一旦加深后模块接口处的测试存在，浅模块上的旧单元测试就变成了废物——删除它们。
- 在加深后模块的接口处编写新测试。**接口就是测试面**。
- 测试断言通过接口的可观察结果，而不是内部状态。
- 测试应该在内部重构后仍然存活——它们描述行为，而不是实现。如果实现变了测试就得改，说明测试在测接口之后的东西。
