# 接口设计

当用户想要为选定的加深候选探索替代接口时，使用此并行子 agent 模式。基于"设计两次"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**模块**、**接口**、**接缝**、**适配器**、**杠杆**。

## 流程

### 1. 框定问题空间

在生成子 agent 之前，为选定的候选编写面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 与该模块相关的 docs/rules/ 规则约束（接口必须落在这些约定之内；默认遵守，确需偏离则显式标出冲突 + 说明理由 + 交用户裁决）
- 它将依赖的依赖项，以及它们属于哪个分类（见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的示意性代码草图来具象化约束——不是提案，只是让约束变得具体的方式

向用户展示，然后立即进入步骤 2。用户在阅读和思考的同时，子 agent 并行工作。

### 2. 生成子 agent

使用 Agent 工具并行生成 3+ 个子 agent。每个必须为加深后的模块产出一个**根本不同的**接口。

为每个子 agent 提供独立的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖分类、接缝后面是什么）。简报独立于步骤 1 中面向用户的问题空间说明。给每个 agent 不同的设计约束：

- Agent 1："最小化接口——目标 1-3 个入口点。最大化每个入口点的杠杆。"
- Agent 2："最大化灵活性——支持多种用例和扩展。"
- Agent 3："为最常见的调用者优化——让默认场景变得极简。"
- Agent 4（如适用）："围绕端口与适配器设计跨接缝依赖。"

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇、CONTEXT.md 词汇，以及与该模块相关的 docs/rules/ 规则，使每个子 agent 的命名既与架构语言一致、也与项目的领域语言一致，并使接口落在项目规则约束之内。

每个子 agent 输出：

1. 接口（类型、方法、参数——加上不变量、顺序、错误模式）
2. 使用示例，展示调用者如何使用它
3. 实现在接缝后面隐藏了什么
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里杠杆高，哪里杠杆薄
6. 与相关 RULES 的符合性说明——该接口默认遵守相关 docs/rules/ 规则；若有偏离，显式标出冲突并说明理由

### 3. 展示和比较

按顺序展示设计，让用户逐个消化，然后用散文进行比较。从**深度**（接口处的杠杆）、**局部性**（变更集中在哪里）和**接缝位置**三个维度对比。

比较之后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果不同设计中的元素可以很好地组合，提出一个混合方案。要有主见——用户想要的是一个有力的判断，而不是一份菜单。
