# 约定式提交（Conventional Commits）

**只做 commit，不做 push。** 提交完成后不要自动推送到远程仓库，由用户自行决定何时 push。

## 提交消息格式

```
<类型>[可选 范围]: <描述>

[可选 正文]

[可选 脚注]
```

## 类型说明

| 类型 | 使用场景 |
|------|---------|
| `feat` | 新增功能 |
| `fix` | 修复 bug |
| `docs` | 仅文档变更 |
| `style` | 格式调整（空格、分号等，不影响逻辑） |
| `refactor` | 重构代码（既不修 bug 也不加功能） |
| `perf` | 性能优化 |
| `test` | 添加或修改测试 |
| `build` | 构建系统或外部依赖变更 |
| `ci` | CI 配置变更 |
| `chore` | 其他不涉及 src 或 test 的变更 |
| `revert` | 回退之前的提交 |

## 破坏性变更

两种标记方式：

1. 在类型/范围后加 `!`：`feat(api)!: 移除已废弃的接口`
2. 在脚注中添加 `BREAKING CHANGE:`：
```
feat: 迁移到新的认证系统

BREAKING CHANGE: JWT token 格式变更，所有客户端必须重新认证
```

两者可以同时使用。脚注中的 `BREAKING CHANGE` 必须大写。

## 工作流程

当用户想要提交时：

### 快速路径优先

当前会话已经说清改了什么，且待提交文件可明确归属于同一个逻辑变更时，
必须走快速路径。不得为重新理解已知改动而默认读取完整 diff、`CONTEXT.md`
或其他项目文档。

1. 在一次预检中获取仓库根目录和 `git status --short`。宿主支持时，
   必须合并执行互不依赖的只读检查，不得串行拆成多轮。
2. 确认范围清晰后，只暂存当前会话相关的文件。“当前会话相关”包括：
   本轮对话中创建或修改的文件、用户明确指定的文件，以及根据任务上下文
   可明确判断属于同一逻辑变更的文件。暂存区为空时不要先询问用户。
3. 暂存后，在一次检查中执行 `git diff --cached --stat` 和
   `git diff --check --cached`。前者用于确认最终范围，后者失败时必须先处理。
   若暂存区仍为空，停止并告知用户没有可提交的改动。
4. 直接根据当前会话、用户原话和文件路径生成提交消息，然后执行
   `git commit`。提交成功后，在一次轻量复核中获取 `git log -1 --oneline`
   和 `git status --short`。只有用户要求详细审计时才读取完整提交详情。

### 风险升级

仅当出现以下任一情况时，才退出快速路径：

- 暂存区包含不属于当前会话的改动，或无法可靠判断文件归属；
- 同一路径存在 staged/unstaged 混杂改动；
- 可能误提交凭据、本地配置、个人数据、大批生成物或二进制文件；
- 当前会话不足以判断变更类型、影响范围或是否存在破坏性变更。

升级后只补充当前决策所需的信息：

1. 按文件读取相关精确 diff，不得无条件展开整个仓库的 diff。
2. 只有提交消息中的领域术语无法从当前会话、用户原话或代码命名确定时，
   才按 `docs/agents/domain.md` 定位并读取相关 `CONTEXT.md`。找不到时继续提交，
   使用代码路径、包名、文件名或用户原话中的命名，不得臆造术语。
3. `commit` 阶段不得因术语缺失主动扩大为 `CONTEXT.md` 更新；仅当用户明确
   要求补充术语，或当前任务本就包含该文档变更时才更新。
4. 如果范围仍不清、可能混入用户改动或无法确认破坏性影响，必须在
   执行 `git add` 或 `git commit` 前使用提问工具确认。

按以下规则生成提交消息：

- **标题行**：`<类型>(<范围>): <描述>` — 祈使语气，小写开头，不加句号，不超过 72 字符
- **术语**：优先沿用当前会话和用户原话中已确定的命名；风险升级并读取
  `CONTEXT.md` 后，使用其稳定术语，避免同一业务对象出现多套叫法
- **正文**（如需解释变更）：说明变更了什么以及为什么，每行不超过 72 字符
- **脚注**：如有破坏性变更写 `BREAKING CHANGE:`，关联问题写 `Refs: #123`，关联提交写 `Refs: abc1234`（不带 #），协作者信息等

## 编写规则

- 标题行不超过 72 字符
- 描述使用祈使语气（"添加"而非"添加了"，"修复"而非"修复了"）
- 描述小写开头（英文时）
- 标题行末尾不加句号
- 标题、正文、脚注之间用空行分隔
- 正文解释"做了什么"和"为什么"，而非"怎么做的"
- 一个提交对应一个逻辑变更 — 如果变更跨越多个关注点，建议拆分

## 范围选择

范围应反映受影响的模块、组件或领域，从以下维度选取：
- 目录名：`feat(auth):`、`fix(api):`
- 组件名：`style(button):`、`test(user-service):`
- 领域概念：`feat(计费):`、`fix(国际化):`、`feat(订单):`、`fix(权限):`

当变更范围较广或无法归入单一模块时，省略范围。

## 示例

简单 bug 修复：
```
fix(订单): 处理空订单号时不再崩溃
```

带正文的新功能：
```
feat(物料): 添加模糊搜索支持

使用 Levenshtein 距离进行近似物料名称匹配。
可通过 `search.fuzzyThreshold` 选项配置阈值。
```

破坏性变更：
```
feat(合同)!: 审批接口返回值使用信封格式包装

BREAKING CHANGE: 合同审批 API 响应现在包装在 `{ data, meta }` 对象中，
不再直接返回数组。

Refs: #456
```

文档变更：
```
docs: 为新开发者添加贡献指南
```

回退提交：
```
revert: feat(物料): 添加模糊搜索支持

Refs: abc1234
```

多脚注：
```
fix(库存): 防止并发扣减时的竞态条件

引入请求 ID 追踪机制，丢弃过期响应。

Reviewed-by: Alice
Refs: #123
```
