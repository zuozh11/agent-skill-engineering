# 约定式提交（Conventional Commits）

**只做 commit，不做 push。** 提交完成后不要自动推送到远程仓库，由用户自行决定何时 push。

## 提交消息格式

```
<类型>[可选 范围]: <描述>

[可选 正文]

[可选 脚注]
```

## 类型说明

| 类型 | 使用场景 |
|------|---------|
| `feat` | 新增功能 |
| `fix` | 修复 bug |
| `docs` | 仅文档变更 |
| `style` | 格式调整（空格、分号等，不影响逻辑） |
| `refactor` | 重构代码（既不修 bug 也不加功能） |
| `perf` | 性能优化 |
| `test` | 添加或修改测试 |
| `build` | 构建系统或外部依赖变更 |
| `ci` | CI 配置变更 |
| `chore` | 其他不涉及 src 或 test 的变更 |
| `revert` | 回退之前的提交 |

## 破坏性变更

两种标记方式：

1. 在类型/范围后加 `!`：`feat(api)!: 移除已废弃的接口`
2. 在脚注中添加 `BREAKING CHANGE:`：
```
feat: 迁移到新的认证系统

BREAKING CHANGE: JWT token 格式变更，所有客户端必须重新认证
```

两者可以同时使用。脚注中的 `BREAKING CHANGE` 必须大写。

## 工作流程

当用户想要提交时：

### 快速路径优先

默认走快速路径，目标是尽快完成安全的本地 commit。只有提交边界不清、
存在同一路径的 staged/unstaged 混杂改动、疑似敏感/本地配置/大批生成物、
或需要确认破坏性变更时，才展开完整 diff 分析或询问用户。

1. 执行 `git status --short`，并用 `git rev-parse --show-toplevel` 确认仓库根目录。
   若有暂存内容，优先用 `git diff --cached --stat` 快速确认范围。
   不要默认执行完整 `git diff` / `git diff --cached`；只在需要判断具体语义、
   排除混杂改动或生成提交正文时再读取相关文件的精确 diff。
2. 提交前加载目标仓库的 `CONTEXT.md`：
   - 默认读 `docs/CONTEXT.md`。若存在 `docs/CONTEXT-MAP.md`（多 Context monorepo），据本次改动涉及的路径定位相关 Context、读其 `docs/CONTEXT.md`（可能多个），定位细节见 `docs/agents/domain.md`。
   - 生成提交范围和提交消息时必须使用 `CONTEXT.md` 中的领域术语、模块名和业务对象名。
   - 如果找不到 `CONTEXT.md`，继续提交流程，但不要臆造业务术语；使用代码路径、
     包名、文件名或用户原话中的命名，并在结果中提示缺少术语来源。
   - 如果 `CONTEXT.md` 存在但没有覆盖本次变更涉及的业务对象、模块或关键命名，
     优先考虑补充术语；只有本次改动不适合扩大到文档更新时，才只使用代码路径、
     包名、文件名或用户原话中的命名。
3. 默认直接暂存并提交当前会话相关的改动，不要因为暂存区为空就先询问用户。
   “当前会话相关”包括：本轮对话中你创建或修改的文件、用户明确指定的文件、
   以及能够从本轮任务上下文清楚判断属于同一个逻辑变更的文件。
4. 如果暂存区已有变更，先判断它们是否属于当前会话或用户明确要求的提交范围：
   - 属于同一范围：保留暂存内容，并把当前会话相关的未暂存改动一并暂存后提交。
   - 不属于同一范围或判断不清：不要擅自混入提交，使用提问工具确认提交范围。
5. 如果存在未暂存或未跟踪文件，默认只 `git add` 当前会话相关的文件。
   只有在无法可靠判断哪些文件属于当前会话、同一路径存在用户未要求的混杂改动、
   或可能把敏感/本地配置/大批生成物误提交时，才询问用户。
6. 用最小必要信息分析变更内容：
   - 已暂存或即将暂存哪些文件
   - 属于哪种类型（功能、修复、重构等）
   - 影响范围应优先采用 `CONTEXT.md` 中的领域术语；没有对应术语时再用目录名、
     组件名或包名，并判断是否应先补充 `CONTEXT.md`
   - 是否包含破坏性变更；不确定且风险真实存在时再深入 diff 或询问
7. 提交前确认最终暂存内容：
   - 默认用 `git diff --cached --name-only` 或 `git diff --cached --stat` 确认文件范围。
   - 只有需要判断具体语义、破坏性变更或混杂改动时，才读取相关文件的完整
     `git diff --cached -- <file>`。
   - 执行 `git diff --check --cached`。如果失败，先修复空白/冲突标记等问题再提交。
8. 按以下规则生成提交消息：
   - **标题行**：`<类型>(<范围>): <描述>` — 祈使语气，小写开头，不加句号，不超过 72 字符
   - **术语**：标题、正文和范围优先使用 `CONTEXT.md` 的稳定术语，避免同一业务对象出现多套叫法
   - **正文**（如需解释变更）：说明变更了什么以及为什么，每行不超过 72 字符
   - **脚注**：如有破坏性变更写 `BREAKING CHANGE:`，关联问题写 `Refs: #123`，关联提交写 `Refs: abc1234`（不带 #），协作者信息等
9. 使用生成的消息执行 `git commit`。提交成功后，用 `git log -1 --oneline` 和
   `git status --short` 做轻量复核；只有用户要求详细审计时才输出完整提交详情。

## 编写规则

- 标题行不超过 72 字符
- 描述使用祈使语气（"添加"而非"添加了"，"修复"而非"修复了"）
- 描述小写开头（英文时）
- 标题行末尾不加句号
- 标题、正文、脚注之间用空行分隔
- 正文解释"做了什么"和"为什么"，而非"怎么做的"
- 一个提交对应一个逻辑变更 — 如果变更跨越多个关注点，建议拆分

## 范围选择

范围应反映受影响的模块、组件或领域，从以下维度选取：
- 目录名：`feat(auth):`、`fix(api):`
- 组件名：`style(button):`、`test(user-service):`
- 领域概念：`feat(计费):`、`fix(国际化):`、`feat(订单):`、`fix(权限):`

当变更范围较广或无法归入单一模块时，省略范围。

## 示例

简单 bug 修复：
```
fix(订单): 处理空订单号时不再崩溃
```

带正文的新功能：
```
feat(物料): 添加模糊搜索支持

使用 Levenshtein 距离进行近似物料名称匹配。
可通过 `search.fuzzyThreshold` 选项配置阈值。
```

破坏性变更：
```
feat(合同)!: 审批接口返回值使用信封格式包装

BREAKING CHANGE: 合同审批 API 响应现在包装在 `{ data, meta }` 对象中，
不再直接返回数组。

Refs: #456
```

文档变更：
```
docs: 为新开发者添加贡献指南
```

回退提交：
```
revert: feat(物料): 添加模糊搜索支持

Refs: abc1234
```

多脚注：
```
fix(库存): 防止并发扣减时的竞态条件

引入请求 ID 追踪机制，丢弃过期响应。

Reviewed-by: Alice
Refs: #123
```
