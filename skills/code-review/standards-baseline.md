# 通用评审基线

项目明确记录的规范始终优先。以下条目是需要结合上下文判断的风险信号，不是硬性违规；只有在当前 diff 中造成真实理解、正确性或维护成本时才报告。

- **含义不明的名称**：名称无法说明其持有的数据或承担的行为。
- **重复代码**：相同逻辑形态在本次变更中重复出现，容易产生不一致。
- **Feature Envy**：方法主要操作另一个对象的数据，职责可能放错位置。
- **Data Clumps**：同一组字段或参数反复成组传递，可能缺少明确领域类型。
- **Primitive Obsession**：字符串或数字承载了需要独立约束的领域概念。
- **重复分支**：围绕同一类型或状态的判断在多处重复，容易产生行为漂移。
- **Shotgun Surgery**：一个逻辑变化迫使大量无关位置同步修改，变化没有被集中。
- **Divergent Change**：同一模块因多个无关原因反复变化，职责混杂。
- **Speculative Generality**：为当前需求不存在的未来场景增加抽象、参数或扩展点。
- **Message Chains**：调用方依赖过长的对象导航链，泄漏内部结构。
- **Middle Man**：模块主要透传调用，没有隐藏复杂度或形成有效接口。
- **Inheritance Misuse**：继承关系要求子类忽略或覆盖大部分父类行为，组合可能更合适。

报告时必须指出当前 diff 中的具体证据和实际影响。仅凭个人偏好、理论可能性或脱离变更范围的旧代码不得形成 finding。
