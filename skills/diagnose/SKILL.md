# Diagnose

一套针对疑难 bug 的纪律。只有在明确说明理由时才能跳过阶段。

探索代码库时，使用项目的领域术语表来建立相关模块的清晰心智模型，并查阅相关项目规则：

- **`docs/rules/`** — 项目规则（RULES）。这不是"建立心智模型的背景阅读"，而是诊断与修复时必须遵守的项目级约束：后续的假设、插桩、修复都不得与之冲突。按 `docs/agents/domain.md` 的布局定位规则（多 Context 含系统级与相关 Context 级两层——两层都要读，别只读根目录），依据文件名（自描述，说明每条规则管什么）判断哪些与当前任务相关，读取相关规则；拿不准就读。

## 阶段 1 — 建立反馈循环

**这就是核心技能。** 其他一切都是机械操作。如果你有一个快速、确定性的、agent 可运行的 pass/fail 信号来检测 bug，你就能找到原因——二分法、假设检验和插桩都只是消费这个信号。如果你没有，再怎么盯着代码看也救不了你。

在这里投入不成比例的精力。**要激进。要有创意。拒绝放弃。**

### 构建反馈循环的方法——大致按此顺序尝试

1. **失败测试**——在能触及 bug 的任何接缝处：单元、集成、端到端。
2. **Curl / HTTP 脚本**——对运行中的开发服务器发请求。
3. **CLI 调用**——用固定输入，将 stdout 与已知正确的快照做 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，对 DOM/控制台/网络做断言。
5. **回放捕获的 trace。** 将真实的网络请求/载荷/事件日志保存到磁盘；在隔离环境中通过代码路径回放。
6. **一次性测试工具。** 启动系统的最小子集（一个服务，mock 依赖），用单个函数调用来执行 bug 代码路径。
7. **属性/模糊循环。** 如果 bug 是"有时输出错误"，运行 1000 个随机输入寻找失败模式。
8. **二分工具。** 如果 bug 出现在两个已知状态之间（commit、数据集、版本），自动化"在状态 X 启动，检查，重复"以便 `git bisect run`。
9. **差异循环。** 对同一输入分别通过旧版本和新版本（或两种配置）运行并 diff 输出。
10. **HITL bash 脚本。** 最后手段。如果必须由人来点击，用 `scripts/hitl-loop.template.sh` 来驱动*他们*，使循环仍然结构化。捕获的输出反馈给你。

建立正确的反馈循环，bug 就已经解决了 90%。

### 迭代循环本身

把循环当作产品来对待。一旦你有了*一个*循环，问：

- 能更快吗？（缓存 setup，跳过无关初始化，缩小测试范围。）
- 能让信号更清晰吗？（断言具体症状，而不是"没崩溃"。）
- 能更确定性吗？（固定时间，设定随机种子，隔离文件系统，冻结网络。）

一个 30 秒的不稳定循环几乎和没有循环一样差。一个 2 秒的确定性循环是调试超能力。

### 非确定性 bug

目标不是干净的复现，而是**更高的复现率**。循环触发 100 次，并行化，加压力，缩小时间窗口，注入 sleep。一个 50% 复现率的 bug 是可调试的；1% 不行——持续提高复现率直到可调试。

### 当你确实无法建立循环时

停下来明确说明。列出你尝试过的方法。向用户请求：(a) 访问能复现的环境，(b) 捕获的制品（HAR 文件、日志转储、core dump、带时间戳的录屏），或 (c) 允许添加临时生产环境插桩。**不要**在没有循环的情况下继续假设。

在你确信循环可靠之前，不要进入阶段 2。

## 阶段 2 — 复现

运行循环。观察 bug 出现。

确认：

- [ ] 循环产生的是**用户**描述的失败模式——不是恰好在附近的另一个失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中可复现（或者对于非确定性 bug，复现率足够高以支持调试）。
- [ ] 你已捕获确切症状（错误消息、错误输出、慢响应时间），以便后续阶段可以验证修复确实解决了它。

在复现 bug 之前不要继续。

## 阶段 3 — 假设

在测试任何假设之前，生成 **3-5 个排序的假设**。单假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 会使 bug 消失 / <改变 Z> 会使它更严重。"

如果你无法陈述预测，这个假设只是直觉——丢弃或锐化它。

**在测试之前向用户展示排序列表。** 他们通常有领域知识可以立即重新排序（"我们刚部署了对 #3 的改动"），或知道他们已经排除的假设。低成本的检查点，大幅节省时间。不要阻塞等待——如果用户不在就按你的排序继续。

## 阶段 4 — 插桩

每个探针必须映射到阶段 3 中的一个具体预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**——如果环境支持。一个断点胜过十条日志。
2. **定向日志**——在区分假设的边界处。
3. 永远不要"记录所有东西然后 grep"。

**给每条调试日志打标签**，使用唯一前缀，如 `[DEBUG-a4f2]`。最后清理变成一次 grep。无标签的日志留存；有标签的日志删除。

**性能分支。** 对于性能回退，日志通常是错误的工具。改为：建立基线测量（计时工具、`performance.now()`、profiler、查询计划），然后二分。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在修复**之前**编写回归测试——但前提是存在**正确的接缝**。

正确的接缝是指测试能在调用点处执行**真实的 bug 模式**的位置。如果唯一可用的接缝太浅（bug 需要多个调用者但只有单调用者测试，单元测试无法复现触发 bug 的调用链），在那里写回归测试只会给人虚假的信心。

**如果不存在正确的接缝，这本身就是发现。** 记录下来。代码库架构正在阻止 bug 被锁定。将此标记给下一阶段。

如果存在正确的接缝：

1. 将最小化的复现转化为该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. **校验修复符合相关 RULES。** 绿灯但违反约束视为未完成——回去重做。确需偏离某条规则时，不要静默违反：显式标出冲突、说明理由，交用户裁决。
6. 对原始（未最小化的）场景重新运行阶段 1 的反馈循环。

## 阶段 6 — 清理 + 事后分析

宣布完成前必须完成：

- [ ] 原始复现不再复现（重新运行阶段 1 的循环）
- [ ] 回归测试通过（或接缝缺失已记录）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（`grep` 前缀）
- [ ] 一次性原型已删除（或移至明确标记的调试位置）
- [ ] 最终正确的假设已在 commit / PR 消息中陈述——以便下一个调试者学习

**然后问：什么能预防这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、纠缠的调用者、隐藏的耦合），将具体信息交给 `/improve-codebase-architecture` skill。在修复完成**之后**提出建议，而不是之前——你现在比开始时掌握了更多信息。

**若 bug 暴露出一条代码里看不见、下个人会再踩的约束**（隐式不变量、调用顺序要求、必须成对出现的操作），提议把它固化为 `docs/rules/` 下的一条 RULES（落到系统级还是相关 Context 级见 `docs/agents/domain.md`），让约束从此显式可查。
