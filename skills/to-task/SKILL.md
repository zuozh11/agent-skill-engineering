# To Task

将需求拆解为可独立实现、可独立验证的任务卡。普通需求使用 vertical slice；无法保持每个切片独立可用的宽范围机械改造使用 expand-contract。

**任务卡永远的核心是：实现者拿到任务卡即可直接动手。**

为此，`to-task` 必须前置完成必要的需求澄清、代码调研、主要入口定位和方案选择，收口业务、架构与接口层面的关键决策。实现者不需要回头翻需求、追问业务口径或重新决定主要实现方案。

任务卡是一份详细的方案设计，不是提前写好的代码。它应明确主要改动入口、核心处理流程和复用能力，但不编写实现代码，也不穷举可由实现过程自然确定的辅助文件、内部方法和代码行。

需求来源不限：PRD 文档、当前对话上下文、口头描述、截图、task 等均可作为输入。

任务文件存放在 `docs/scratch/<NN>-<中文需求名称>/tasks/` 下。

## 流程

### 1. 明确需求边界

确认输入来源并提取核心需求：

- 如果是文档（PRD/task），直接读取。
- 如果是对话上下文，归纳用户已表达的意图。
- 如果输入存在歧义或关键信息缺失，调用 `grill-with-docs` 与用户对齐。

只有需求目标、业务口径和验收方向足以支持方案设计时，才进入后续步骤。

### 2. 加载领域知识

按需查阅以下资源：

- **领域文档**：按 `docs/agents/domain.md` 定位并读取相关 `CONTEXT.md` 与 RULES。只要某条 RULES 有哪怕约 1% 的可能影响当前任务，就必须读取；只有明确完全无关时才可跳过。相关文档不存在则静默继续。

任务标题和方案描述使用 `CONTEXT.md` 中的统一术语。

### 3. 调研并收口方案

基于代码与领域事实，完成足以让实现者直接动手的调研和方案设计。按需读取相关 Skill，核对现有实现与项目惯例，但不为制作任务卡而穷举文件、内部调用或代码细节。

每张任务卡应明确与本任务相关的内容：

- **为什么要改**：简要说明本任务解决的问题，详细背景留在 PRD。
- **当前如何工作**：说明现有流程、行为和能力缺口。
- **目标行为是什么**：明确完成后系统应提供的端到端行为。
- **在哪里实施**：判断新增还是修改现有能力，并明确应在哪个接口、流程、模块或扩展点实施；能够从仓库确认时可以写出核心类或方法，但不穷举辅助修改点。
- **采用什么方案**：写清主要实现路径，以及不采用其他路径时需要遵守的边界。
- **复用哪些能力**：指出必须调用或复用的现有服务、组件、数据结构和状态定义。
- **数据如何变化**：说明涉及的字段、状态、接口契约和持久化语义。
- **按什么顺序处理**：说明关键步骤及其先后关系。
- **如何验收**：给出可以独立执行的操作或请求，以及可观察的预期结果。

无法确认主要方案，或存在影响实施的业务口径、字段、状态、接口契约及 RULES 冲突时，不写占位方案，使用 `提问工具` 与用户对齐，每题给出推荐选项。

### 4. 选择拆分策略

先判断本次属于普通需求还是宽范围重构。

#### 普通需求：vertical slice

将需求拆成 **tracer bullet** 任务。每个任务必须：

- 端到端切穿所有必要层次；
- 完成后可独立验证；
- 自身包含足够完整的方案，拿到即可直接动手；
- 优先形成多个薄 slice，而不是少数厚 slice；
- 没有真实依赖和预计写集冲突时保持彼此独立，不人为串成依赖链。

`Blocked by` 只表示真实业务依赖或可预期的共享代码冲突。无依赖时写空数组；有依赖时每项必须包含准确的任务文件名和串行原因。任务编号和实现习惯不能作为依赖关系。

```yaml
Blocked by:
  - task: 01-创建公共接口.md
    reason: 后续任务依赖该接口
```

#### 宽范围重构：expand-contract

宽范围重构是一次机械性变化的影响面横跨大量调用方，例如共享字段改名、公共模型换型或基础接口迁移，导致任何单个 vertical slice 都无法独立保持系统可用。跨模块开发本身不等于宽范围重构；只要仍能交付端到端行为，就继续使用 vertical slice。

确认属于宽范围重构后，按以下阶段拆分：

1. **Expand**：先引入新结构或新接口，并保留旧形式的兼容路径，使现有调用方继续工作。
2. **Migrate**：按模块、包或影响面拆成多个迁移任务。每个任务都依赖 Expand，并明确本批次迁移范围和独立验证方式。
3. **Contract**：所有迁移任务完成后，删除旧结构、兼容路径和过渡代码；该任务依赖全部 Migrate 任务。

如果单个迁移批次确实无法独立验证，在任务卡中明确共同集成点、串行原因和最终集成验证任务，不得伪装成彼此独立的 slice。

### 5. 写入任务文件

按依赖顺序（blocker 先写入）在 `docs/scratch/<NN>-<中文需求名称>/tasks/` 下创建文件。

- 有 PRD 时，必须从 PRD 的父目录派生 `tasks/` 路径，不重新猜测需求目录名称。
- 没有 PRD 时，用户指定了已有需求目录则沿用；否则扫描 `docs/scratch/` 直属目录中匹配“开头连续数字 + 连字符”的前缀，取最大值加一作为 `NN`，从 `01` 开始，至少两位零填充，并以需求主题的中文名称创建目录。中文需求名称优先使用 `CONTEXT.md` 中的统一术语，不得改写为英文 slug 或拼音。历史无编号目录不迁移、不参与计数；创建目录前重新扫描，发生目录碰撞时取下一个编号重试。

文件名使用 `NN-<中文任务名称>.md`，两位零填充并按依赖顺序编号，例如 `01-创建数据表.md`、`02-新增查询接口.md`。中文任务名称必须使用 `CONTEXT.md` 中的统一术语，并准确概括该任务的端到端目标，不得改写为英文 slug 或拼音；编号继续供下游 `impl` 读取和引用。

## task 文件模板

```markdown
---
Status: ready-for-agent
Blocked by: []
---

## Parent

<有 PRD：[需求标题](../PRD.md)；从口头、截图等无 PRD 输入拆解时，写一句话需求摘要或指向真实存在的需求来源>

## What to build

<本任务解决什么问题，以及完成后用户或系统能获得什么端到端行为>

## 适用 RULES

- <RULE-NN：规则标题——本任务必须遵守的执行约束>
- <如无适用规则：None，已扫描并排除 docs/rules/ 中明确完全无关的规则>

## 现状

<当前流程或能力如何工作、缺少什么；只写理解本方案所需的代码事实，不展开无关实现细节>

## 方案设计

### <实施位置> — <本改动的目标>

<明确在哪个接口、流程、模块或扩展点实施；能够从仓库确认时可写出核心类或方法，不穷举辅助修改点>

1. <核心处理步骤；自然写清相关的能力复用和数据变化>
2. <核心处理步骤>
3. <核心处理步骤>

## 验收

- 场景 1：<操作或请求> → <可观察的预期结果>
- 场景 2：<边界或失败场景> → <可观察的预期结果>
```

## 完成标准

将任务卡交给未参与需求讨论的实现者：如果其无需回看 PRD、询问业务口径或重新选择主要方案，能够从任务卡指明的位置直接开始实施，则任务卡合格。

以下情况不合格：

- “在保存接口里加上编号生成逻辑，按需求要求处理。”
- “增加认证初始化，具体怎么接入实现时再看。”
- 只复述需求，把实施位置、接入方式或主要路径留到实现时再决定。
- 留下未确认的字段、状态、接口契约或业务规则。
- 反过来穷举文件和行号，或编写方法体、代码 diff、伪代码及其他实现代码。

## 方案设计粒度示例

```markdown
### `SupplierAppService.save` — 注册供应商后初始化认证记录

修改现有供应商保存流程，不新增对外接口。

1. 在供应商基本信息落库成功后初始化认证记录。
2. 初始化逻辑复用 `CertificationService.initialize`，不直接操作 Mapper。
3. 只有首次创建供应商时执行，编辑已有供应商时不得重复初始化。
4. 供应商保存与认证初始化使用同一事务。
5. 初始化失败时整体回滚。
6. 不改变现有接口的请求和响应结构。
```

这个示例已经收口主要入口、复用能力和处理顺序，执行者可以直接动手；
