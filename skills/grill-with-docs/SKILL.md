<what-to-do>

针对这个方案的每个方面进行不留情面的追问，直到我们达成共识。把方案映射成一棵设计树：每个决策都是一个节点，依赖该决策的后续决策挂在它的下游。

追问前先区分事实与决策：

- **事实**：能够通过代码、配置、现有文档、日志或其他环境证据确认的内容。先自行查证并给出证据，不把调查转交给用户。
- **决策**：需要在多个可行选择之间确定业务口径、范围、权衡或目标的内容。即使仓库存在先例，也必须向用户提问并等待选择；先例只作为推荐依据，不能替用户决定。

### 分轮批量追问

1. **构建设计树。** 列出当前已知的决策、依赖关系和尚未触及的分支。不必向用户展开完整内部树，但必须用它约束提问顺序。
2. **计算当前前沿。** 前沿是所有“前置决策已经确定，现在即可回答”的未决问题。只有前沿问题可以进入本轮。
3. **先查证事实。** 前沿问题如果依赖环境事实，使用代码库、文档、工具或子 Agent 自行查证。可并行查证的事实就并行处理；正在查证的事实视为未满足的前置，只暂停它的下游分支，不阻塞其他已就绪的前沿问题。
4. **整批提问。** 使用当前宿主的提问工具，在一轮中提出尽可能多的前沿问题。为每个问题编号，给出互斥选项，并把推荐选项放在首位、说明理由。如果提问工具限制单轮问题数，按工具上限分批；未问到的前沿问题留在后续批次，不得默认答案。
5. **等待整批回答。** 用户回答前不继续追问本批的下游决策。某个问题如果依赖本轮另一个尚未回答的问题，它属于后续轮次，不得混入本轮。
6. **重算前沿。** 用每轮答案更新设计树，推进已解锁的下游分支，然后重复上述流程。

前沿为空只表示设计树已走完。此时必须再用提问工具请用户确认是否已达成共识。只有用户明确确认，追问才算完成，后续 PRD、任务拆解或实现流程才能继续。对话中已经逐项确认的术语和规则仍按下文即时写入；未确认的决策不得落盘。

</what-to-do>

## 领域感知

自行查证问题时，同时查找已有的文档：

### 文件结构

领域文档的布局（单 Context / 多 Context）、读取与落盘解析，统一见 `docs/agents/domain.md`。本 skill 据此定位并即时更新 `CONTEXT.md` 与 RULES：多 Context（有 `docs/CONTEXT-MAP.md`）先据地图定位相关 Context 的 `CONTEXT.md`；RULES 无论布局都写入根 `docs/rules/`。

按需创建文件——只在有内容要写时才创建。如果 `CONTEXT.md` 不存在，在第一个术语确定时创建。如果 `docs/rules/` 不存在，在需要第一条 RULES 时创建。

## 对话过程中

### 追问侧重

根据仓库技术栈调整追问的重心：
- 后端仓库：围绕业务规则发力——状态流转是否完备、持久化边界是否清晰、查询隔离是否合理、流程回写是否有遗漏等
- 前端仓库：围绕交互行为发力——页面流转是否闭环、组件状态归属是否明确、表单校验规则是否穷尽、接口调用契约是否对齐等

### 挑战术语表

当用户使用的术语与 `CONTEXT.md` 中已有的术语冲突时，立即指出。"你的术语表将'cancellation'定义为 X，但你似乎指的是 Y——到底是哪个？"

### 锐化模糊术语

当用户使用含糊或多义的术语时，提出一个精确的规范术语。"你说的'account'——是指 Customer 还是 User？这是两个不同的东西。"

### 讨论具体场景

当讨论领域关系时，用具体场景来压力测试。构造探测边界情况的场景，迫使用户精确定义概念之间的边界。

### 与代码交叉验证

当用户陈述某个东西如何工作时，检查代码是否一致。如果发现矛盾，指出来："你的代码取消的是整个 Order，但你刚才说可以部分取消——哪个是对的？"

### 与已有 RULES 交叉验证

当用户方案触及已有 RULES 覆盖的领域时，检查是否违反。如有冲突，立即指出并要求对齐或显式说明为何偏离："RULES 规定金额一律以分为单位，但你的方案用了元——改方案还是更新这条规则？"

### 即时更新 CONTEXT.md

当一个术语被确定时，立即更新 `CONTEXT.md`（多 Context 下写哪个 Context 的 `CONTEXT.md` 见 `docs/agents/domain.md`）。不要攒着——确定一个就记录一个。使用 `docs/agents/context-format.md` 中的格式。

`CONTEXT.md` 应该完全不含实现细节。不要把 `CONTEXT.md` 当作规格说明、草稿本或实现决策的存放处。它是术语表，仅此而已。

### 何时提议 RULES

当一条规则项目长期有效、agent 在相关任务中不遵守就会实现跑偏时，提议把它记录为 RULES。它可以是两类之一：

- **决策类**（架构、选型、边界）——同时满足：难以逆转、缺上下文会让人困惑、有真实权衡。
- **约定/约束类**（错误码统一包装、金额以分为单位、日志规范、命名约定）——只要项目长期有效、agent 必须遵守即可，**不要求**难逆转或多方案权衡。

一次性的、局部的、能从代码一眼看出的，跳过。格式见 `docs/agents/rules-format.md`。
