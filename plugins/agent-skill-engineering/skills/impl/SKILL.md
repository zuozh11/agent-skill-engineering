# Impl

根据任务卡或当前对话实现代码变更，两种输入方式地位相同。

**原子提交是本 Skill 最核心的原则：每个 commit 只表达一个可独立理解和回滚的变更。**

## 流程

开始前检查并记录主工作区中已有的已暂存、未暂存和未跟踪改动。工作区不 clean 本身不阻塞实现；已有改动视为用户所有，不得自动 stash、覆盖、暂存或提交。仅当已有改动与本次范围重叠，且无法可靠地分离或归属本次改动时，才暂停并请用户处理。

### 1. 确定实现单元

从用户提供的输入中确定本次要完成的独立实现单元：

- **任务卡输入**：读取指定的 task 文件，理解目标、现状、方案和验收条件，并按 `ready-for-agent → completed → archived` 维护状态。
- **对话输入**：从当前对话提取目标、范围、已确定的方案和验收方式；上下文足够时直接实现，不要求先生成任务卡。

如果一次包含多个可独立完成的改动，先划分实现单元，后续分别实现和提交。只有缺少的信息会真正阻塞实现时，才调用 `grill-with-docs` 与用户对齐。

以任务卡为输入时，先检查执行资格：

- 只有 `Status: ready-for-agent` 的任务可以进入调度；`completed` 任务不重复实现
- 存在 `Blocked by` 时，读取其中引用的任务卡；只有全部依赖均已完成，当前任务才解锁
- 依赖引用不存在、状态异常或形成循环时，停止对应任务并报告
- 一次输入多个任务时，只调度当前已解锁的任务，其余任务等待依赖集成

### 2. 选择执行方式

每次执行都必须调用当前宿主的提问工具，简要告知实现单元数量和依赖情况，并请用户选择：

- **主线程直接实现**：主 Agent 在当前工作区中按依赖串行实现，不创建子 Agent、worktree 或临时分支。适合单点小改或希望快速完成的任务。
- **隔离 worktree 子 Agent 实现**：按依赖调度子 Agent，每个实现单元使用独立 worktree 和临时分支，由主 Agent 集成。适合多任务隔离或并行实现。

不得根据任务大小代替用户选择，也不得因为用户在请求中表达了偏好而跳过本次确认。选定后全程保持该方式；除非用户重新选择，不得自动切换或降级。当前宿主没有提问工具时，暂停并在普通对话中请用户选择。

### 3. 加载领域知识与编码约束

使用以下领域资源：

- **领域文档**：按 `docs/agents/domain.md` 定位并使用相关 `CONTEXT.md`；所有项目规则均必须遵守。相关 `CONTEXT.md` 不存在则静默继续。

### 4. 按选定方式实现

#### 主线程直接实现

- 主 Agent 在当前工作区内实现、检查和提交，不调度子 Agent
- 多个实现单元按依赖顺序串行处理；无依赖时按任务卡编号或输入顺序处理
- 每完成一个实现单元，先交付其原子 commit，再继续下一个

#### 隔离 worktree 子 Agent 实现

主 Agent 按依赖关系分批调度实现单元：

- 同一批已解锁任务并行执行；存在依赖的任务在依赖集成后进入下一批
- 发现任务卡遗漏的明显共享代码冲突时，仅将相关任务降级为串行，不修改任务卡
- 每个实现单元交给一个子 Agent，并使用独立 worktree 和临时分支；串行任务的 worktree 必须从前一个任务集成后的最新分支创建
- 无法使用子 Agent 或安全创建 worktree 时，说明原因并暂停，由用户决定是否改选主线程直接实现

子 Agent 必须收到完整的实现目标；存在 task 文件时，同时提供文件路径和内容、`CONTEXT.md` 指针，并明确所有项目规则均必须遵守。

#### 共同实现与提交规则

- 严格遵守所有项目 RULES；存在 task 文件时，同时重点关注其中列出的 RULES
- 根据任务卡或对话确定的目标查看现有代码，将已确定的方案映射到实际代码结构，并识别必要的修改点
- 不重新选择任务卡或对话中已经收口的主要方案；若代码事实证明方案不可行或互相矛盾，停止实现并明确报告，而不是自行改换方案
- 遵守 `CLAUDE.md` / `AGENTS.md` 中的编码指南和规则
- 匹配实现范围内相关的 Skill
- 不添加实现范围之外的功能、重构或抽象
- 每次提交前只暂存当前实现单元的改动，并检查 staged diff；已有改动不得进入本次 commit
- 隔离模式下，子 Agent 只处理分配的任务范围，并交付原子 commit
- 一个任务默认形成一个 commit；确有多个可独立理解和回滚的变更时，可以交付有序的 commit 列表
- 隔离模式下，子 Agent 必须返回全部 SHA 及其目的，不得交付 WIP、fixup 或其他过程性提交
- 发现实现超出任务范围、改变既定方案或引入新的任务依赖时，停止并交回主 Agent 重新拆分
- 隔离模式下，子 Agent 不修改任务卡状态

### 5. 集成与状态流转

主线程模式在每个实现单元的原子 commit 完成后直接进入状态流转，不执行 cherry-pick、worktree 或临时分支操作。

隔离模式下，同一批任务结束后，主 Agent 按任务卡编号依次处理；无任务卡时按实现单元顺序处理。

- 按子 Agent 返回的顺序完整 cherry-pick 一个任务的全部 commits；全部成功后，该任务才视为集成完成
- cherry-pick 冲突时中止本次集成并保留 worktree，交回原执行子 Agent 基于最新主分支处理；冲突来自主工作区相关未提交改动时，暂停并请求用户处理
- 单个任务失败不影响无依赖的成功任务；失败任务只阻塞依赖它的后续任务
- 集成成功后移除 worktree 和临时分支
- 未成功集成的 worktree 和临时分支必须保留，并报告路径、分支和失败原因

两种模式共同遵守：

- 实现单元成功提交或集成后，由主 Agent 将对应任务改为 `completed`
- 同一 `tasks/` 目录下全部任务均为 `completed` 时，由主 Agent 统一改为 `archived`

## 硬性规则

- **不扩展 scope**：只做任务卡或对话确定的改动，发现相邻问题只提醒不修
- **不过度抽象**：匹配现有代码风格，不引入任务未要求的设计模式
- **积极复用**：优先复用现有能力、组件和同类实现，不重复实现已有逻辑，也不为复用扩大实现范围
- **不猜 API**：不确定的框架 API 先 grep 现有代码中的用法，再查 references
