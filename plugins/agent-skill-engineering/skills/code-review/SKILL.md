# Code Review

对代码变更执行只读评审，分别回答两个问题：

- **Standards**：变更是否符合仓库记录的编码约束、项目规则和高置信度代码质量基线？
- **Spec**：变更是否完整、准确地实现了 PRD、任务卡或当前对话确定的需求？

两个维度必须独立调查、独立报告，不能让一个维度的结论掩盖另一个维度的问题。本 Skill 只报告问题，不修改代码；用户后续明确要求修复时再进入实现流程。

## 流程

### 1. 固定评审范围

先把评审范围解析为可重复执行的 diff：

- 用户指定 commit、branch、tag 或 range 时，直接使用并先验证引用真实存在。
- 用户指定 PR 编号或 URL 时，优先使用可用的代码托管连接器或 CLI 只读解析其 base/head 与 patch；当前宿主无法读取时明确报告，不根据编号猜测范围。
- 用户明确要求评审未提交改动时，同时读取 `git diff` 与 `git diff --staged`。
- 用户要求评审当前分支但未指定基点时，先检查当前分支、上游分支和默认分支；能够明确确定基点时使用 merge-base，不能明确确定时使用提问工具确认。

分支范围默认使用 `git diff <fixed-point>...HEAD`，并记录 `git log <fixed-point>..HEAD --oneline`。引用无效或 diff 为空时立即停止并报告，不启动后续评审。

### 2. 定位需求来源

按以下顺序查找本次变更的需求依据：

1. 用户明确指定的 PRD、任务卡、Issue、PR 或当前对话目标；
2. commit 消息或变更文件中引用的任务卡；
3. 任务卡 `Parent` 指向的 `PRD.md`；
4. 与当前需求目录、分支名称或变更主题明确对应的本地 PRD/任务卡。

读取真实存在的完整内容，不根据文件名猜测需求。如果仍无需求来源，继续 Standards 评审，并在最终结果中把 Spec 标为“无可用需求依据”。

### 3. 加载项目规范

读取与变更范围相关的规范来源：

- 当前范围生效的 `AGENTS.md`、`CLAUDE.md` 或等效 Agent 指令；
- 按 `docs/agents/domain.md` 定位的 `CONTEXT.md`，以及全部项目 RULES；
- 仓库内真实存在的编码规范、测试约定和模块约束；
- [standards-baseline.md](./standards-baseline.md) 中的通用评审基线。

项目明确记录的规则优先于通用基线。工具已经能够确定性检查的问题交给工具，不重复制造人工风格意见。

### 4. 独立执行两个评审维度

宿主支持子 Agent 时，必须并行安排两个只读子 Agent：

- **Standards Agent**：接收固定 diff、commit 列表、项目规范来源和通用基线，只检查规范违反与代码质量风险。
- **Spec Agent**：接收相同 diff、commit 列表和完整需求来源，只检查需求遗漏、错误实现、scope creep 与验收缺口。

两个子 Agent 不得修改文件，也不得读取对方的中间结论。宿主不支持子 Agent 时，按相同输入顺序执行两个独立检查，并保持证据与结论分栏，不混写推理。

如果没有需求来源，跳过 Spec Agent；如果没有代码 diff，两个维度都不执行。

### 5. 聚合结果

按以下结构输出：

```markdown
## Standards

- [P1] <问题标题> — `<文件:行>`
  - 证据：<违反的项目规则或命中的基线>
  - 影响：<为什么会造成真实风险>
  - 建议：<最小修复方向>

## Spec

- [P1] <问题标题> — `<文件:行>`
  - 需求依据：<PRD/任务卡中的对应条目>
  - 偏差：<遗漏、错误、scope creep 或验收缺口>
  - 建议：<最小修复方向>
```

严重级别使用 `P0` 到 `P3`：`P0` 阻断交付或造成重大事故，`P1` 高概率导致错误行为，`P2` 局部缺陷或明显维护风险，`P3` 低风险改进。没有发现时明确写“未发现问题”，不要为了填充报告制造意见。

## 完成标准

- diff 范围和基点明确且可复现；
- Standards 与 Spec 使用各自真实来源，独立完成；
- 每条问题都包含紧凑文件位置、证据、影响和最小修复方向；
- 不把推测写成缺陷，不报告工具已经覆盖的纯格式问题；
- 全程保持只读，没有修改工作区、提交或远端状态。
