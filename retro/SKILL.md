# Retro

复盘 review 修改，闭合工程管线的反馈环。

## 调用方式

```
/retro <commit-range>
```

示例：
- `/retro HEAD~5..HEAD`
- `/retro abc123..HEAD`
- `/retro main..feature-branch`

## 流程

### 1. 确定分析范围

从用户指定的 commit range 获取变更。同时检查工作区：

- `git diff <range>` — 已提交的变更
- `git diff` + `git diff --staged` — 未提交的改动（一并纳入分析）

### 2. 加载上下文

| 来源 | 用途 |
|------|------|
| 当前会话对话 | 提取 review 讨论中的认知盲区（B 层） |
| task 文件 | 对比预期 vs 实际（A 层） |
| PRD | 理解原始需求意图 |
| CONTEXT.md | 检查术语是否需要更新 |
| docs/adr/ | 检查是否有新的架构决策需要记录 |

如果 task 文件或 PRD 不存在（用户直接从口头描述开始 impl），跳过 A 层对比，只做 B 层分析。

### 3. A 层分析：预期 vs 实际

对比 task 文件中的预期改动与实际代码变更：

- 哪些预期改动没做？为什么？
- 哪些实际改动超出了 task 描述？是 review 驱动的还是实现中发现的？
- 验收条件是否全部满足？偏差在哪里？

### 4. B 层分析：认知盲区

从当前会话的 review 对话中提取：

- reviewer 指出了哪些 grill/PRD 阶段没考虑到的问题？
- 哪些修改暴露了 CONTEXT.md 中术语定义的不精确？
- 哪些修改反映了未记录的架构约束？
- 哪些修改是 agent 反复犯的模式性错误？

### 5. 分类与标注

将每条发现分类为三种产物之一：

| 分类 | 触发条件 | 产物 |
|------|----------|------|
| 术语更新 | CONTEXT.md 中的定义需要补充或修正 | CONTEXT.md 更新 |
| 架构决策 | 发现了 hard-to-reverse 的决策且未记录 | 新增/更新 ADR |
| 行为规则 | 发现了 agent 应遵守的约束或模式 | 规则建议 |

为每条发现标注置信度：
- **高**：明确的模式性问题，大概率会反复出现
- **中**：有一定通用性，但可能是特定场景
- **低**：可能是一次性问题，但值得记录

### 6. 批量审批

将所有发现以清单形式呈现给用户，每条包含：

- 分类标签（术语更新 / 架构决策 / 行为规则）
- 置信度（高 / 中 / 低）
- 内容摘要
- 建议的 draft 内容

使用 AskUserQuestion 让用户批量审批：
- 对于术语更新和 ADR：用户确认后直接写入对应文件
- 对于行为规则：用户确认内容后，再询问写入位置

### 7. 写入

根据用户审批结果执行写入：

**CONTEXT.md 更新**：直接编辑现有 CONTEXT.md，补充或修正术语定义。遵循 CONTEXT.md 现有格式。

**ADR**：在 `docs/adr/` 下新建文件，编号递增。遵循项目现有 ADR 格式。

**行为规则**：根据用户指定的位置写入：
- 项目 guides skill（如果存在）
- CLAUDE.md
- 用户指定的其他位置

## 规则写入位置策略

检查项目是否存在 guides 类 skill 或项目级规则文件：
1. 查找 `.claude/` 下是否有 guides/rules 相关配置
2. 查找项目根目录是否有 CLAUDE.md
3. 将候选位置列出，由用户在审批时选择

## 硬性规则

- **不修改本 skill 集**：retro 不更新 agent-skill-engineering 仓库中的任何 skill 文件
- **不自动写入**：所有产物必须经过用户审批才能写入
- **不扩大范围**：只分析用户指定的 commit range，不回溯更早的历史
- **不编造偏差**：如果 review 修改合理且无认知盲区，如实报告"无显著发现"
- **使用 CONTEXT.md 术语**：分析和产出中使用项目已定义的术语
